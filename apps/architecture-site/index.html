MomentMarkt

A city wallet that surfaces a perfectly relevant local offer the moment it fits — generated by an LLM, bounded by the merchant, never authored as static copy.

Stack: React Native + Expo · React + Vite · FastAPI · Pydantic AI · Azure OpenAI gpt-5.5 Cities: Berlin Mitte (39 OSM merchants) · Zurich HB (30) Spec: work/SPEC.md (spec-v04, ready)
Section 01

The Wedge

Competitors do personalized offers: the merchant authors a static offer string, the system distributes it, the LLM routes who sees what. MomentMarkt does generative offers within merchant-set bounds: the merchant authors no offer copy at all — they set bounds, and an LLM generates the actual offer in real time when an agent decides conditions match.

Section 03

Agent Topology

Five strategic agents define the architecture. Three ship today; one (Negotiation) ships as a module with tests but waits on v2 wiring; one (Bounds Manager) is fixture-baked today and replaces fixtures with a portal in v3. Three supporting Pydantic AI agents handle per-card emotional copy, headline contextualisation, and menu validation during onboarding.

Strategic agents

AgentRoleStatusModule
Opportunity Given context, what offer should this merchant make? Drafts offer + GenUI widget spec. shipped opportunity_agent.py
Surfacing Given wrapped user signals, should we fire this approved offer now? Deterministic scoring; LLM rewrites headline only on the fired card. shipped surfacing_agent.py
Preference Given swipe + dwell history, re-rank cross-merchant candidates. Anchor pinned at index 0; permutation-validated. shipped preference_agent.py
Negotiation Given user reaction, adjust discount within bounds. discount_pct always inside [floor, ceiling] by heuristic and a final post-LLM clamp. module shipped · wiring v2 negotiation_agent.py
Bounds Manager What constraints did this merchant set? Fixture-baked today; merchant portal owns it in v3. v3 — merchant portal fixtures today

Supporting agents

AgentRoleTriggerModule
Subhead Per-card emotional one-liner ("Cocoa weather. Eight stools, three taken."). Cached by (merchant_id, weather_trigger, time_bucket). Every card in the swipe stack subhead_agent.py
Headline Rewrite Contextualises copy_seed against the wrapped user context and high-intent boost. One LLM call per fired offer. Only when Surfacing fires llm_agents.py
Menu Edit Validates and refines OCR-extracted menu items during merchant onboarding stage 3. Onboarding pipeline menu_edit_agent.py

Why Negotiation matters

It enforces a hard invariant: discount_pct is always inside [discount_floor_pct, discount_ceiling_pct] — both by the deterministic heuristic and by a final clamp on the LLM output, so any hallucination cannot regress the contract. "User rejected −5% three times, dwelled 1.4s on −15% last visit → next offer is −18%, capped by merchant's −25% floor."

Two views, four surfaces. The 2-view IA refactor (Discover · Browse) puts the swipe stack at full screen as the default — but the underlying surfaces are still the four roles below. None is removable; each answers a real user question.

Section 05

Merchant Surfaces

The merchant side is the inverse of the consumer side: instead of receiving generated offers, the merchant onboards their menu and hours, then bounds the offers an agent can make on their behalf. Today's web dashboard ships four sections plus a five-stage live onboarding pipeline driven by Pydantic AI menu extraction.

Onboarding — the speed-run merchant cut

Onboarding is the opposite of campaign work: ingest what the merchant already has and turn it into structured rules. Five stages, each backed by a real backend route under /onboarding/{id}/…:

  1. Drop stage 1
    Merchant enters URL or business name. FlowIntroStep.tsx kicks off an onboarding session via POST /onboarding.
  2. Menu read stage 2
    Real OCR runs on the menu image. MenuReadStep.tsx polls GET /onboarding/{id}/status while the pipeline extracts categories and items.
  3. Menu confirm stage 3
    Merchant edits items in a chat-style flow with the Menu Edit Agent. MenuConfirmStep.tsx ↔ POST /onboarding/{id}/menu/agent.
  4. Hours stage 4
    HoursPicker.tsx writes through to POST /onboarding/{id}/hours. Pre-populated from public sources where possible.
  5. Processing stage 5
    A demand-curve preview (typical vs. live) appears alongside suggested bounds. POST /onboarding/{id}/complete finalizes the merchant.

Dashboard sections

Today live

Daily briefing — pending-offer count chip, live demand curve + ROI strip, active offer mini-cards with inventory burndown rings. Polls /merchants/{id}/summary every 2s.

Today.tsx

Offers live

Inbox of drafted + approved offers. Per-offer headline, discount, status badge (pending_approval · approved · auto_approved · rejected). One-tap approve/reject; auto-approve rule hints surfaced inline.

Offers.tsx

Bounds scaffolded

Discount floor/ceiling sliders, category chips, opening hours, blackout windows, brand tone. The visual proof of the wedge. Today: fixture-baked. v3: portal-driven.

Bounds.tsx

Settings scaffolded

Sparkasse account binding, integration keys, basic merchant profile. Cosmetic, present so the dashboard reads as a real product.

Settings.tsx

Future — mobile companion (v3)

The structural symmetry with the consumer wallet: push when offers fire under bounds, quick-glance status, swipe-to-block in the audit log. The merchant's swipe surface mirrors the consumer's. Not decoration — it is the merchant's escape hatch from a system that otherwise generates on their behalf 24/7.

The eight beats

  1. Discover open 0:00–0:05

  2. Lens switch + right-swipe 0:05–0:12

Intent tokens (the locked enum)

The wrapped enum carries one of: lunch_break.cold, commute.evening_rushed, tourist.midday_browsing, weekend_wander, late_night.solo. h3_cell_r8 is a coarse ~1 km hex; the candidate pool is the user's cell + 1 ring. weather_state is a derived enum, not raw forecast data. high_intent is a boolean flag composed on-device from active screen time, map foreground, and coupon-browse proxies.

The wedge compounds into a privacy story that is structural, not theatrical. Because the merchant authors bounds rather than offers, and the LLM generates within those bounds based on on-device signals, the system that maximises offer relevance also minimises centralised user data. Consent banners are a substitute for architecture; this is the architecture.

Section 10

Eight Design Principles

Durable invariants that govern how MomentMarkt curates, ranks, and surfaces offers. Non-negotiable for v1 and v2. If a future feature threatens any of them, the right response is to scope-cut the feature, not the principle.

  1. The list is reality. The swipe is curation. MerchantSearchList shows every merchant in the city's catalog by simple distance — no LLM filtering, no personalization, no paid placement. The swipe re-orders; it never removes.
  2. No paid placement. The LLM's prompt has no "boost merchant X" instruction. Ranking is driven by distance, swipe history, offer attributes, and live signals. If sponsored ever lands, it is explicitly marked and annotates rather than replaces.
  3. Preferences stay on-device. In production, dwell time, swipe direction, and inferred intent never leave the phone. Only {intent_token, h3_cell_r8} reaches the backend. Structural defense against centralised manipulation.
  4. Always offer a non-personalized fallback. The "Nearby" lens is pure distance — no LLM, no swipe-history influence. The antidote to magician's force: the user always has a verifiable algorithm.
  5. Reasoning is inspectable. Long-press a swipe card → reveal the matched signals. Opaque ranking is not acceptable. Every NegotiatedOffer carries a reasoning string for the audit log.
  6. The LLM is one of several mechanisms. For-you = LLM. Best-deals = deterministic sort. Right-now = rule-based. Nearby = pure distance. The user opts into a mechanism; nothing is a black box that runs everything.
  7. Text search is text-match only. Drawer search bar is a plain substring filter over display_name, category, neighborhood. No LLM re-ranking, no "smart" interpretation. What you type is what you get.
  8. Don't break expectations between sessions. A left-swipe doesn't make a merchant disappear from the list. Swipe history influences ranking within the swipe stack. City swap (Berlin ↔ Zurich) resets swipe history — different cultural context, different semantics.
